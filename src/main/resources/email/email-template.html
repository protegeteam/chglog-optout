As part of our work on ontology engineering we intend to produce a publicly available dataset of WebProtégé change history logs. To build this dataset, we would like to include the change history logs for the WebProtégé projects that you own. To preserve anonymity, any logs that you allow us to include will be de-identified and obfuscated so that they do not contain any projects names, user names, dates & times, or ontology content. While it is unlikely that you'll directly benefit from this, you will be contributing to science that will advance knowledge about ontology engineering. You may indirectly benefit from improvements to tools that come about due to analysis of the published dataset by researchers in the field. If you are happy for your logs to be included in this dataset then you do not need to do anything else. If you do not want some or all of your change history logs to be included in this dataset then you must opt out. There is a link at the end of this email that will take you to an opt out form. If you decide to opt out then you must complete the opt out form on or before March 18th. The rest of this email provides more details about our goals and how we de-identify and obfuscate logs to preserve anonymity.

Why do we want to do this?

While many ontologies are published online, the change histories for ontology projects are not usually available. This hampers researchers who are interested in ontology engineering research areas such as ontology evolution, ontology versioning, incremental reasoner development and optimisation, and query engine optimisation. Many of these research areas do not require actual identifying domain details that are pertinent to an ontology project, they only require logical and structural changes. Thus, having de-identified, obfuscated change history logs may be of great benefit to the field of ontology engineering research. We believe that general enough conclusions could be drawn from research into these logs so that it will be possible to make novel tool workflow and performance improvements.

What is a change history log?

As you may know, WebProtégé tracks the changes made to a project so that they can be reviewed and collaborators can see who did what and when they did it. The list of changes in a project is known as the project change history log (or simply a log). A portion of the change history for one of the Protégé team’s projects, Aero – “An ontology of commercial aviation”, is show below. This portion includes the changes for the class “L1011”.

As you can see, a plain change history log records the dates and times of when changes were applied, the user names of the authors and the axioms that were added or removed. According to the WebProtégé terms and conditions, this information is stored securely on the WebProtégé server and we will never publish plain change history logs like this. We would, however, like to publish de-identified, obfuscated versions of your change history logs. The de-identified, obfuscated version of the change history log above is shown in the figure below.

A de-identified, obfuscated change history log:

  • Does not contain WebProtégé project identifiers
  • Does not contain real user names
  • Does not contain real dates or times
  • Does not contain real or meaningful class, property, individual or datatype names
  • Does not contain real or meaningful labels and other annotation values
  • Does not contain real or meaningful language tags
  • Does not contain any real or meaningful data that could be used to identify the domain modelled by the project
  • Only preserves the sequence or logical and structural aspects of the change history

It is important for you to know that it is impossible for users of the dataset to reconstruct the original ontologies, or origninal change history log, from a de-identified, obfuscated change history log.

How we get from a plain change history log to a de-identified, obfuscated change history log is described in detail below.

Where will the dataset be published?

We will publish the dataset online. It is likely to be published so that it is available from a sub-domain of protege.stanford.edu, for example datasets.protege.stanford.edu. This means that anyone will be able to access the dataset.

How many logs will the dataset contain?

We cannot say for sure because this depends on the number of people who opt out. However, if nobody were to opt out we would expect the dataset to contain around 30,000 logs.

Who will have access to the dataset?

Since the dataset will be published online it will be available to anyone. However, we would expect that this dataset is only of interest to researchers who want to use the data for studying the structural aspects of ontology changes.

How do we de-identify and obfuscate a log?

Taking the above plain change history log as an example, to de-identify and obfuscate it,

  1. We replace author user names with alphanumeric identifiers. In this example we replace “M Horridge” with “U1”.
  2. We replace real dates & times with offset timestamps. In this example, August 28th is replaced with 153453257380ms, which represents 1776 days since the project was created.
    Offset timestamps can be used to calculate the number of milliseconds between changes, but cannot be used to identify the actual date or time that the changes took place. In this case, the fact the changes were made on August 28th and August 31st would not be recoverable from this data.
  3. We replace entity IRIs (identifiers) with gibberish/meaningless (but pronounceable) names. In this example, the IRI for “L1011” is replaced with “pirad-mudag” and “skos:altLabel” with “luguv-lajuv”.
    Note that these gibberish names are scoped to a particular change history log. Thus “skos:altLabel” in one project history change history log might be obfuscated as “luguv-lajuv”, but in another project change history log it could be obfuscated as something completely different, like “falab-taduk”. We do not obfuscate IRIs for built-in OWL entities, such as owl:Thing.
  4. We replace all strings, numbers and other scalar value occurrences with alpha-numeric symbols of the form “Vn” where n is a randomly generated hexadecimal encoded integer. In this example “WhisperLiner” is replaced with “V17b4b80d”.
    In a single change history log, every occurrence of “Whisper Liner” is represented by the same alpha-numeric symbol. However, if the string “WhisperLiner” occurs in multiple project change history logs it may have a different alpha-numeric symbol in each de-identified, obfuscated change history log.
  5. We replace all language tags with alphanumeric identifiers. In this example the language tag “@en” is replaced with “@L0”.
    This removes any concrete details of the languages used by a project.
